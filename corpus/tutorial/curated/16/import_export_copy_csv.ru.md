# Импорт и экспорт CSV через COPY и psql в PostgreSQL 16

## Назначение

Этот материал показывает безопасный и воспроизводимый процесс загрузки и выгрузки CSV в PostgreSQL. Он закрывает практические вопросы: когда использовать `COPY`, когда `\copy`, как проверить корректность данных и как избежать типичных ошибок формата.

## Когда использовать

- нужно массово загрузить данные из CSV в таблицу;
- нужно выгрузить результаты SQL-выборки в CSV для обмена;
- требуется быстрый перенос данных между окружениями;
- нужно проверить импорты перед продакшен-загрузкой;
- команда часто сталкивается с ошибками delimiter/header/encoding.

## Простое объяснение

`COPY` выполняется на стороне сервера PostgreSQL и работает с файлами, доступными серверному процессу. `\copy` — meta-command клиента `psql`; файл читается/пишется на стороне клиента. В большинстве учебных и прикладных сценариев проще и безопаснее начинать с `\copy`, потому что не нужен доступ к файловой системе сервера.

Надёжный паттерн: сначала загружать данные в staging-таблицу, делать валидацию и только потом переносить в целевые таблицы с ограничениями. Это уменьшает риск поломки production-данных из-за грязного CSV.

## Предварительные условия

- структура CSV понятна: delimiter, header, quoting, кодировка;
- целевая таблица создана и типы данных подходят;
- есть права на INSERT в целевую таблицу;
- если используется `COPY`, сервер имеет доступ к нужному пути файла;
- есть место для staging-таблицы и проверки качества данных.

## Минимальный рабочий пример

```sql
CREATE TABLE products_staging (
    name text,
    price_text text
);
```

```text
\copy products_staging(name, price_text) FROM 'products.csv' WITH (FORMAT csv, HEADER true)
```

```sql
INSERT INTO products(name, price)
SELECT name, price_text::numeric(10,2)
FROM products_staging;
```

```text
\copy (SELECT name, price FROM products ORDER BY name) TO 'products_export.csv' WITH (FORMAT csv, HEADER true)
```

```sql
COPY products(name, price)
FROM '/var/lib/postgresql/products.csv'
WITH (FORMAT csv, HEADER true);
```

## Пошаговый алгоритм

1. Проверь формат CSV (разделитель, заголовок, кодировку, пустые значения).
2. Создай staging-таблицу с "мягкими" типами для первичной загрузки.
3. Выполни импорт через `\copy` (или `COPY`, если нужен серверный путь).
4. Проверь количество строк и аномальные значения в staging.
5. Преобразуй и перенеси данные в целевую таблицу с явными cast/валидацией.
6. Зафиксируй reject-стратегию для ошибочных строк (журнал, quarantine).
7. Для экспорта используй `\copy (SELECT ...) TO ...` с явным `ORDER BY` при необходимости.
8. Сверь итоговые объёмы и повторяемость процесса на тестовом наборе.

## Как проверить результат

```sql
SELECT count(*) AS staging_rows FROM products_staging;
SELECT count(*) AS target_rows  FROM products;
```

```sql
SELECT *
FROM products_staging
WHERE price_text !~ '^[0-9]+(\.[0-9]+)?$'
LIMIT 20;
```

```text
\copy (SELECT count(*) AS rows_exported FROM products) TO 'products_export_check.csv' WITH (FORMAT csv, HEADER true)
```

- количество импортированных строк совпадает с ожиданием;
- невалидные строки обнаружены и обработаны до загрузки в целевую схему;
- экспортируемый CSV открывается корректно и содержит ожидаемые колонки/порядок.

## Типовые ошибки

- Путать файл клиента и файл сервера (`\copy` vs `COPY`).
- Загружать сразу в боевую таблицу без staging-проверок.
- Игнорировать кодировку и delimiter, особенно при данных из Excel.
- Надеяться на неявные cast и получать частично грязные данные.
- Не фиксировать версию входного файла и правила импорта.

## Безопасность и ограничения

- Серверный `COPY` требует аккуратного контроля прав и путей.
- CSV может содержать чувствительные данные; хранение и транспорт нужно защищать.
- Большие импорты могут влиять на I/O и блокировки, планируй окно работ.
- Для production полезны идемпотентные скрипты и журнал загрузок.

## Короткий вывод

Для большинства сценариев удобнее `\copy`: файл обрабатывается на стороне клиента `psql`, что упрощает доступ и безопасность. Надёжный импорт строится через staging-слой, валидацию и только затем перенос в целевые таблицы. Проверки объёма и формата после загрузки обязательны, иначе CSV-пайплайн становится источником тихих ошибок.

# Основы секционирования таблиц в PostgreSQL 18

## Назначение

Этот материал объясняет, когда partitioning действительно полезен и как запустить базовую схему range-секционирования без лишней сложности. Он закрывает вопросы "зачем делить таблицу на секции", "как проверить, что pruning работает" и "какие ошибки чаще всего допускают на старте".

## Когда использовать

- таблица большая и естественно делится по времени или другому ключу;
- нужны регулярные операции архивирования/удаления старых данных;
- важны предсказуемые запросы по диапазонам ключа секционирования;
- нужно снизить операционные риски при обслуживании больших таблиц;
- требуется подготовить масштабируемую схему для time-series или event-данных.

## Простое объяснение

Partitioning в PostgreSQL даёт одну логическую таблицу, которая физически состоит из дочерних секций. Приложение обычно работает с родительской таблицей, а PostgreSQL направляет записи и запросы в нужные секции.

Главная польза — управление объёмом и ускорение запросов, когда фильтры совпадают с ключом секционирования. Тогда planner может исключить лишние секции (partition pruning). Но partitioning не является "магическим ускорителем" всех запросов: если запросы не фильтруют по ключу, выгода может быть минимальной.

## Предварительные условия

- есть явный ключ секционирования (часто дата/время);
- объём данных и паттерны запросов реально требуют секционирования;
- есть план создания будущих секций заранее;
- есть стратегия индексации для секций и контроля bloat;
- есть окно для миграции данных, если таблица уже в эксплуатации.

## Минимальный рабочий пример

```sql
CREATE TABLE measurements (
    id bigint GENERATED ALWAYS AS IDENTITY,
    measured_at date NOT NULL,
    value numeric(12,3) NOT NULL,
    PRIMARY KEY (id, measured_at)
) PARTITION BY RANGE (measured_at);
```

```sql
CREATE TABLE measurements_2026_01
PARTITION OF measurements
FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');

CREATE TABLE measurements_2026_02
PARTITION OF measurements
FOR VALUES FROM ('2026-02-01') TO ('2026-03-01');
```

```sql
INSERT INTO measurements (measured_at, value)
VALUES ('2026-01-10', 42.000), ('2026-02-11', 41.500);
```

```sql
SELECT tableoid::regclass AS physical_partition, id, measured_at, value
FROM measurements
ORDER BY measured_at;
```

## Пошаговый алгоритм

1. Подтверди, что основная часть запросов фильтруется по одному устойчивому ключу.
2. Выбери тип секционирования (обычно RANGE для дат/времени).
3. Создай parent table и задай ключ секционирования.
4. Создай секции на текущий и ближайшие периоды заранее.
5. Настрой индексы, которые реально нужны в запросах по секциям.
6. Проверь вставку и маршрутизацию строк в нужные партиции.
7. Выполни `EXPLAIN` для типовых запросов и проверь pruning.
8. Организуй регулярную операционную задачу: создавать новые секции и архивировать старые.

## Как проверить результат

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT count(*)
FROM measurements
WHERE measured_at >= DATE '2026-02-01'
  AND measured_at < DATE '2026-03-01';
```

```sql
SELECT tableoid::regclass AS physical_partition, count(*)
FROM measurements
GROUP BY tableoid
ORDER BY physical_partition;
```

- planner должен читать только релевантные секции при фильтре по ключу;
- вставки должны попадать в ожидаемую партицию;
- нагрузка обслуживания (архивация, удаление) должна стать предсказуемее.

## Типовые ошибки

- Секционировать таблицу, которая ещё мала и не требует этого.
- Выбрать ключ, который почти не используется в фильтрах.
- Не создавать секции заранее и получать ошибки при вставке.
- Ожидать ускорения запросов без условий по partition key.
- Игнорировать сопровождение: секции созданы, но процесс их ротации не автоматизирован.

## Безопасность и ограничения

- Partitioning усложняет схему и операционные процедуры; это осознанный trade-off.
- Неудачный ключ секционирования может ухудшить планы и сопровождение.
- Массовые миграции в существующей системе требуют теста на блокировки и время переключения.
- Для production важно иметь rollback-план и скрипты создания/архивации секций.

## Короткий вывод

Секционирование в PostgreSQL полезно, когда данные и запросы естественно подчиняются partition key, обычно по времени. Практическая выгода приходит от pruning и удобства жизненного цикла данных, а не от автоматического ускорения любой выборки. Успешная схема требует регулярного сопровождения секций и проверок планов.
